# ProjectTransfer GUI v2.7

**Исправление v2.2:** все BAT-файлы пересозданы в чистом ASCII с Windows-переносами CRLF. Это устраняет ошибки CMD вида `is not recognized as an internal or external command`, возникавшие из-за UTF-8 и LF в версии 2.1. Для распаковки рекомендуется короткий путь латиницей, например `C:\Automation\ProjectTransfer`.

# Перенос проектов — ускоренная оконная версия

Программа сохраняет исходное API-first ядро переноса проектов между аккаунтами myrosmol.ru и добавляет графический интерфейс, управляемый параллелизм и безопасные автоматические политики.

## Запуск

1. Запустите `install.bat`.
2. Запустите `run.bat`.
3. Выберите `исходник.xls`, `итог.xls`, режим и параметры.

## Что сохранено

Сохранены все исходные режимы: скачивание, перенос, оба этапа, проверка разбора Word, обработка одной строки и лимита, автоматический и ручной выбор проекта, выбор по дате, заполнение существующего черновика или создание нового, API-заполнение разделов, пропуск/повтор проблемного этапа, резервный DOM-режим, диагностика сайта, скриншоты и XLS/CSV-отчёты.

`transfer.py` и `legacy_transfer.py` побайтно совпадают с исходным ядром. Старый интерфейс запускается через `run_legacy.bat`.

## Ускорение

Независимые аккаунты могут обрабатываться параллельно в 1–4 браузерных контекстах. Параллелизм включается только когда не требуется одновременный ручной ввод. При политиках `ask` или ручном выборе проектов GUI автоматически устанавливает один поток.

Для полностью автоматического пакетного переноса используйте:

- «Автовыбор проекта»;
- «Все аккаунты»;
- при ошибке `skip` или `stop`;
- при дубликате `existing` или `new`;
- 2 потока как безопасное значение, 3–4 — только при стабильном соединении.

## Оконный интерфейс

В окне доступны выбор файлов, режим, строка/лимит, число потоков, видимый или скрытый браузер, блокировка тяжёлых ресурсов, политика ошибок и дубликатов, скриншоты, живой журнал, остановка и поле ответа на интерактивные вопросы программы.

## Безопасность

Пароли не выводятся в журнал. В дистрибутив не включены старые браузерные профили, HAR, скриншоты и журналы, поскольку они могут содержать персональные и сессионные данные. API вызывается только после штатной авторизации в браузере; неподтверждённые endpoint’ы не добавлялись.

## EXE

`build_exe.bat` создаёт Windows-сборку и после завершения автоматически открывает правильную папку `dist\ProjectTransfer`. Запускать нужно только `dist\ProjectTransfer\ProjectTransfer.exe` либо `run_built_exe.bat`. Папка `build` содержит временные файлы PyInstaller и не предназначена для запуска; в версии 2.4 она автоматически удаляется после успешной сборки.

Готовая сборка работает в режиме `onedir`: рядом с `ProjectTransfer.exe` обязательно должна оставаться папка `_internal` с `python314.dll` и другими библиотеками. Нельзя переносить или отправлять один EXE отдельно — переносится целиком папка `dist\ProjectTransfer`.

В версии 2.6 `build_exe.bat` устанавливает переносимый Chromium в `dist\ProjectTransfer\ms-playwright`. GUI и worker явно передают `PLAYWRIGHT_BROWSERS_PATH`, поэтому frozen-приложение больше не ищет браузер во временной папке `_MEI…`.

## Исправление сборки EXE (v2.1)

Установщик теперь создаёт виртуальное окружение строго на Python 3.12 (`py -3.12`), автоматически удаляет окружение, созданное другой версией Python, и устанавливает совместимую актуальную ветку PyInstaller 6.x. Сборщик проверяет наличие модуля PyInstaller, а не только папки `.venv`. Для исключения проблем с кодировкой и длинными путями рекомендуется распаковывать комплект в каталог латиницей, например `C:\Automation\ProjectTransfer`.

## Python 3.14 and broken Microsoft Store Python 3.12

Version 2.3 automatically prefers a working Python 3.14, then 3.13, then 3.12. A broken Microsoft Store 3.12 launcher entry is skipped. For Python 3.14 the dependency `PyYAML` is updated to 6.0.3 and PyInstaller is installed from the compatible 6.21+ branch.

For a clean rebuild, delete `.venv`, run `install.bat`, then `build_exe.bat`.

## Исправление v2.4: запуск из папки build

Ошибка `Failed to load Python DLL ... build\ProjectTransfer\_internal\python314.dll` означает, что был запущен промежуточный EXE из папки `build`. Для запуска используется только итоговая папка `dist\ProjectTransfer`. Сборщик теперь проверяет наличие `_internal\python*.dll`, открывает итоговую папку и удаляет временный каталог `build`, чтобы исключить случайный запуск неправильного файла.


## Исправление v2.6: Chromium в frozen worker

Ошибка `Executable doesn't exist at ... _MEI...\playwright\driver\package\.local-browsers` возникает из-за поведения Playwright в PyInstaller: frozen-процесс по умолчанию ищет браузер внутри временной папки. Версия 2.6 задаёт путь явно, устанавливает переносимый Chromium рядом с приложением и добавляет `repair_playwright_browser.bat` для восстановления уже созданной сборки. Запуск готовой программы выполняется через `run_built_exe.bat` либо напрямую после пересборки 2.6.


## Исправление v2.7: кириллица в журнале

GUI и worker принудительно используют UTF-8 через `PYTHONUTF8=1` и `PYTHONIOENCODING=utf-8`. Стандартные потоки frozen worker перенастраиваются до импорта ядра, а GUI читает байтовый поток с резервным распознаванием CP866 и CP1251. Поэтому русские названия проектов, вопросы выбора и пути к отчётам корректно отображаются независимо от кодовой страницы Windows. Для уже собранной версии можно использовать `run_built_exe.bat` из пакета UTF-8 fix без пересборки; для прямого запуска EXE требуется пересборка версии 2.7.
